The Problem
As agents move from text to actions, capability scales faster than governance.

Three execution risks become systemic:
1) Unauthorized system mutation
2) AI sprawl without centralized execution governance
3) Non-auditable agent decisions

Surfit bounds execution into deterministic missions (Waves) governed by policy at runtime, producing verifiable audit records.

Deterministic Execution Boundaries

Execution is constrained to approved missions, policy versions, and lifecycle scope before actions commit.

Policy-Governed Mutations

Mutation-capable operations are evaluated against policy at runtime with explicit commit boundaries and scoped execution context.

Verifiable Auditability

Hash-chained, verifiable, integrity-checked run records tie outcomes to policy lineage and runtime decisions.

Surfit enforces policy checks before tool invocation, isolates workspace per Wave, anchors policy lineage with hashes, gates mutation paths with wave tokens, and exposes audit verification endpoints.

Policy-Governed Graph Executor
Graph-based Wave execution with policy check before every tool invocation. Each node is evaluated at runtime. The execution structure is governed — the reasoning inside tool nodes is not constrained.
02
Governance
Policy Lineage Anchoring
Canonical SHA256 policy hash computed at Wave initiation. Policy snapshot is persisted per Wave with lineage traceability. Run status and governance provenance are written to DB.
03

Governed LLM Integration
LLM treated as a tool node — not a privileged actor. Provider, model, prompt boundary, raw input, sanitized input, and output all logged. Write actions are policy-evaluated, not manually reviewed.
04

Verifiable Log Chain
Hash-chained execution log with prev_hash → event_hash. Dedicated LLM invocations table with payload hashes. Programmatic integrity verification detects first mismatch and flags post-run mutation.
05

Automated Tamper Testing
CLI tamper simulation with automated pytest validation. Demonstrates integrity failure on mutated rows. DB-backed, cryptographically anchored execution lineage — not UI-level logging.

Cross-Agent Interface
OpenClaw and proprietary internal agents initiate Waves through Surfit’s governed runtime interface. Surfit enforces policy, mutation boundaries, and verification semantics consistently across frameworks.
07

Wave Atlas
The Wave Atlas is Surfit's registry of versioned, governed execution patterns. Each entry defines policy version, tool allowlist, write constraints, and lifecycle scope. Agents request Waves from the Atlas before acting — the Atlas defines what execution is permitted, not how agents think.

Technical FAQ

What is Surfit?

Surfit is a governance runtime for AI agents. It enforces deterministic policy boundaries between agent frameworks and enterprise systems.

Does Surfit replace agent frameworks?

No. Surfit sits in the execution path. Agents can still run in any framework; Surfit governs what those agents are allowed to execute.

What is a Wave?

A Wave is a bounded execution container that carries a pinned policy manifest, execution constraints, and audit lineage for a specific agent action.

Is Surfit tied to a specific model or framework?

No. Surfit is agent-agnostic and designed to work with multiple frameworks and proprietary agent systems.

Does Surfit execute the AI agent?

No. The agent executes normally within its framework. Surfit governs the execution boundary when the agent attempts to interact with enterprise systems.

What happens when a policy violation occurs?

Surfit denies the action and records a tamper-evident governance artifact explaining the reason for the rejection.
